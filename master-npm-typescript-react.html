Node.js & npm

Chapter 01 · Runtime, package management, project structure

What is Node.js?

JavaScript was originally designed to run only inside web browsers — it could manipulate the DOM, respond to clicks, and talk to servers, but it couldn't do anything outside that browser sandbox. Node.js changed everything. Released in 2009, Node.js is a JavaScript runtime built on Chrome's V8 engine that lets you run JavaScript anywhere: on your laptop, a server, inside automated scripts, or as part of a build pipeline.

This is important because almost every modern frontend tool you'll encounter — Vite, Webpack, TypeScript, ESLint, Jest — is itself a Node.js program. When you type npm run dev, you're using Node.js to start a local development server. You're not writing server-side applications, but you're depending on Node.js as the engine that powers your tools.

Node.js also ships with npm (Node Package Manager) built in. Think of npm as an app store for code libraries: over two million open-source packages are available, and you can install any of them into your project with a single command.

package.json — Your Project's Passport

Every npm project has a file called package.json at its root. This is the single source of truth for your project: it records your project name and version, lists every library your project depends on, and defines shortcut commands (called scripts) you can run. Think of it as your project's passport and instruction manual combined.

You create it by running npm init -y in an empty folder. The -y flag means "yes to all defaults" — skip the interactive wizard. Once created, npm will automatically update this file whenever you install or remove packages.

package.json — annotated examplejson
{
  "name": "my-app",           // project name (lowercase, no spaces)
  "version": "1.0.0",        // your app's version
  "scripts": {               // shortcuts you can run with "npm run ..."
    "dev": "vite",            // npm run dev  → starts the dev server
    "build": "vite build",    // npm run build → creates production files
    "test": "vitest"          // npm run test  → runs your tests
  },
  "dependencies": {         // ← packages your app needs IN PRODUCTION
    "react": "^18.3.0",
    "react-dom": "^18.3.0",
    "axios": "^1.7.0"
  },
  "devDependencies": {      // ← packages used ONLY during development
    "typescript": "^5.4.0",
    "vite": "^5.2.0",
    "eslint": "^8.57.0"
  }
}

dependencies vs devDependencies

This distinction trips up many beginners, but it's simple: dependencies are packages your application needs to actually run in production — things like React, which renders your UI, or axios, which makes HTTP requests. devDependencies are tools you only need while you're developing — things like TypeScript (which compiles away before shipping), ESLint (which checks your code), and Vite (your dev server and build tool).

In practice, the difference matters for deployment size and build optimization. Your production bundle only includes dependencies. When a user visits your app, they never download TypeScript itself — it's already been compiled away. Use npm install react for production packages and npm install -D typescript for dev tools.

dependencies

Code your application needs to run in the browser or server. Shipped to your users.

npm install react

Tools that help you build and check your code. Never shipped to users.

Semantic Versioning — What Those Numbers Mean

Every package version follows a pattern: MAJOR.MINOR.PATCH (e.g. 18.3.1). A MAJOR bump means breaking changes — code that worked before might not work after. A MINOR bump adds new features while staying backward-compatible. A PATCH bump fixes bugs without changing any API.

The ^ symbol before a version (called a "caret") tells npm: "install this version or any newer version that doesn't change the major number." So "react": "^18.3.0" means "give me React 18.anything, but don't jump to React 19." This lets you safely receive bug fixes and new features automatically without risking a breaking upgrade.

*Wildcard"*"Any version. Avoid in production — too risky.

node_modules and Why You Never Commit It

When you run npm install, npm creates a folder called node_modules/ and downloads every dependency (and their dependencies, and their dependencies' dependencies...) into it. This folder can easily contain thousands of files and hundreds of megabytes — even for small projects.

This is why you never commit node_modules to Git. Instead, you commit your package.json and package-lock.json, and anyone who clones your repo just runs npm install to regenerate the folder. The package-lock.json is crucial here — it records the exact version of every single package that was installed, so everyone on the team gets identical dependencies, not just similar ones.

.gitignoretext

npm vs yarn — Which Should You Use?

Both npm and yarn do the same job: they install packages from the registry, track versions, and run scripts. Yarn was created by Facebook in 2016 because npm was slow and had some reliability issues at the time. Since then, npm has caught up significantly and is now very fast and reliable.

For learning, use npm — it ships with Node.js, requires no extra installation, and 99% of tutorials use it. The commands are almost identical; the only real differences are syntax details like yarn add vs npm install. You can switch in 10 minutes once you understand the concepts.

TypeScript

Chapter 02 · Static typing, interfaces, generics, utility types

The Problem TypeScript Solves

JavaScript is a dynamically typed language — meaning the type of a variable (string, number, object…) is determined at runtime, not when you write the code. This makes JavaScript very flexible and quick to write, but it creates a whole category of subtle bugs that only appear when your code is actually running, sometimes in production, sometimes with specific user input.

Imagine you write a function that expects a user object and reads user.name. If someone accidentally passes a number instead of an object, JavaScript just says undefined — no warning, no error, just silently wrong behavior. TypeScript catches this the moment you write it, before you even save the file.

TypeScript is a superset of JavaScript — meaning every valid JavaScript file is also valid TypeScript. You gradually add types on top of the code you already know. It compiles down to plain JavaScript before running in browsers, so there's no performance cost at runtime — types only exist at development time.

❌ JavaScript — bug discovered at runtime 
function greet(name) {
  return 'Hello, ' + name.toUpperCase();
}

greet(42);
// 💥 TypeError at runtime
// name.toUpperCase is not a function

greet();
// 💥 TypeError at runtime
// Cannot read properties of undefined
✅ TypeScript — caught while you type 
function greet(name: string): string {
  return 'Hello, ' + name.toUpperCase();
}

greet(42);
// ❌ Error: Argument of type 'number'
//    is not assignable to type 'string'

greet('Alice'); // ✅ "Hello, ALICE"

Primitive Types — The Building Blocks

You add types to variables with a colon after the variable name: let age: number = 25. TypeScript has three primitive types that mirror JavaScript's: number for all numeric values (integers and decimals), string for text, and boolean for true/false. You also have arrays, written as number[] or Array<number>.

One of TypeScript's smartest features is type inference — you don't always have to write the type explicitly. When you write let city = 'Berlin', TypeScript sees the string value and automatically infers that city is a string. If you later try to assign city = 42, TypeScript will catch the error even though you never wrote : string. Only add type annotations when inference can't figure it out — like empty arrays or function parameters.

primitive-types.tsts
// Explicit type annotations
let age: number = 25;
let name: string = 'Alice';
let isAdmin: boolean = false;

// Arrays
let scores: number[] = [95, 87, 92];
let names: string[] = ['Alice', 'Bob'];

// Type inference — TypeScript figures it out from the value
let city = 'Berlin';   // inferred as string — no annotation needed
city = 42;             // ❌ Error: Type 'number' not assignable to 'string'

// Union types — value can be one of several types
let id: string | number;
id = 'abc-123';  // ✅
id = 42;          // ✅
id = true;        // ❌ not in the union

// Literal union types — only specific values allowed
type Status = 'loading' | 'success' | 'error';
let status: Status = 'loading';  // ✅
status = 'pending';              // ❌ 'pending' is not in the type

Interfaces — Describing the Shape of Objects

Most of the data in real applications is structured as objects: a user with a name and email, a product with a price and category, an API response with data and a status code. Interfaces are how TypeScript describes the shape an object must have. They're like a contract: if you say a variable has type User, TypeScript will ensure it has all the required properties with the correct types.

Properties marked with ? are optional — they can be present or absent without causing an error. Properties marked with readonly can be set when the object is first created but can never be changed afterwards. Interfaces can also extend other interfaces, inheriting all their properties — perfect for creating Admin as a more specific version of User.

interfaces.tsts
interface User {
  id: number;                         // required
  name: string;                        // required
  email: string;                       // required
  age?: number;                        // optional — can be omitted
  readonly createdAt: Date;           // set once, cannot be changed
  role: 'admin' | 'user' | 'guest';   // literal union
}

// Extending — Admin inherits all User properties, then adds more
interface Admin extends User {
  permissions: string[];
  lastLogin: Date;
}

// Using the interface — TypeScript checks all required fields
const user: User = {
  id: 1, name: 'Alice', email: 'alice@dev.com',
  createdAt: new Date(), role: 'user'
};
user.createdAt = new Date(); // ❌ Error: cannot assign to readonly

Generics — Types That Work With Any Type

Generics solve a fundamental problem: what if you want to write a function that works with any type of data, but you still want TypeScript to know and check what type it's working with? The naive solution is to use any — but that turns off type checking entirely and you lose all the benefits of TypeScript.

Generics let you write a type placeholder (conventionally called T) that gets filled in with the real type when the function is called. It's like a blank that gets filled in. The key insight is: you write the logic once, TypeScript applies the correct types each time it's used. This is especially useful for API response wrappers, utility functions like filter or find, and custom React hooks.

generics.tsts
// Problem: using 'any' loses all type information
function firstItem(arr: any[]): any {
  return arr[0];
}
const x = firstItem([1, 2, 3]); // x is 'any' — no autocomplete, no safety 😕

// Solution: use a generic type parameter T
function firstItem<T>(arr: T[]): T {
  return arr[0];
}
const num = firstItem([1, 2, 3]);       // num is 'number' ✅
const str = firstItem(['a', 'b', 'c']); // str is 'string' ✅

// Real-world use: typed API response wrapper
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

// Used with different data shapes — T is replaced each time
type UserResponse  = ApiResponse<User>;      // data: User
type PostsResponse = ApiResponse<Post[]>;    // data: Post[]

// Generic async fetch function
async function fetchApi<T>(url: string): Promise<T> {
  const res = await fetch(url);
  return res.json() as T;
}
const user = await fetchApi<User>('/api/user/1'); // user is User ✅

Utility Types — Transforming Existing Types

TypeScript includes a set of built-in "utility types" that transform existing types into new ones. Instead of rewriting similar interfaces over and over, you derive new types from existing ones. For example, you might have a full User interface, but when creating a new user the id doesn't exist yet — Omit<User, 'id'> gives you a version without it. This keeps your types in sync automatically when you update the base interface.

Utility TypeWhat it doesExample result
Partial<T>Makes all properties optionalGood for update/patch operations
Required<T>Makes all properties required (removes ?)When you need every field filled in
Pick<T, K>Keep only the listed propertiesPick<User, 'id' | 'name'>
Omit<T, K>Remove the listed propertiesOmit<User, 'password'>
Readonly<T>Prevents any mutationConfig objects, constants
Record<K, V>Creates a dictionary/map typeRecord<string, User>

React

Chapter 03 · Components, JSX, hooks, state, props, rendering

What is React and Why Does It Exist?

Before React, building interactive UIs meant manually manipulating the DOM — finding elements by ID, updating their text, adding and removing CSS classes. This gets complicated fast. You have to track what state your app is in, decide which parts of the UI need to change, and write the code to change them. With 10 features, this is manageable. With 100, it becomes a bug-ridden mess.

React introduces a fundamentally different mental model: your UI is a function of your data (state). You describe what the UI should look like given some data, and React handles making the real DOM match that description. When data changes, React figures out the minimal set of DOM updates needed and applies them automatically. You stop thinking about "how do I update this button" and start thinking "what does the button look like when isLiked is true?"

React uses a Virtual DOM — a lightweight JavaScript object that represents the real DOM. When state changes, React creates a new virtual DOM, compares it to the previous one (called "diffing"), and only updates the parts of the real DOM that actually changed. This makes updates very efficient.

Components — The Building Blocks

Everything in React is a component — a JavaScript function that accepts some input (called props) and returns JSX describing what should appear on screen. Components can be as small as a single button or as large as an entire page. The power is in composing them: a ProductCard uses a Button and a Badge; a ProductList renders many ProductCards; an App holds a Header, a ProductList, and a Footer.

Component names must start with a capital letter. This is how React distinguishes them from regular HTML elements — <button> is HTML, <Button> is your component. Each component is independent and reusable: you can render <UserCard name="Alice" /> and <UserCard name="Bob" /> side by side, and they each manage their own state independently.

components.tsx — composing components togethertsx
// Each component is a self-contained function
function Badge({ text }: { text: string }) {
  return <span className="badge">{text}</span>;
}

interface UserCardProps {
  name: string;
  role: string;
  isOnline?: boolean;  // optional prop
}

function UserCard({ name, role, isOnline = false }: UserCardProps) {
  return (
    <div className="user-card">
      <h3>{name}</h3>
      Badge text={role} />     // using Badge inside UserCard
      {isOnline && <span>🟢 Online</span>}   // conditional rendering
    </div>
  );
}

// Parent composes children — note capital letter names
function App() {
  return (
    <div>
      <UserCard name="Alice" role="Engineer" isOnline={true} />
      <UserCard name="Bob"   role="Designer" />
    </div>
  );
}

JSX — Writing HTML Inside JavaScript

JSX looks like HTML, but it's actually JavaScript. When you write <h1>Hello</h1> inside a function, Babel/TypeScript transforms it into React.createElement('h1', null, 'Hello') before it runs. This transformation happens automatically — you never see it.

The key difference from HTML: use className instead of class (since class is a reserved word in JavaScript), use htmlFor instead of for, and self-close empty elements with />. JavaScript expressions go inside {curly braces} — you can put any valid JavaScript expression there: variable references, ternary operators, function calls, and more. You cannot use regular if/else statements directly, but you can use ternaries or the && short-circuit operator for conditional rendering.

jsx-rules.tsx — common patterns explainedtsx
function App() {
  const username = 'Alice';
  const isLoggedIn = true;
  const messageCount = 5;

  return (
    <div className="app">                     // className, not class!

      // Embed any JS expression in { }
      <h1>Welcome, {username}!</h1>
      <p>2 + 2 = {2 + 2}</p>

      // Ternary for if/else rendering
      {isLoggedIn
        ? <p>You are logged in ✅</p>
        : <p>Please log in</p>
      }

      // && for "show only if truthy"
      {messageCount > 0 && <span>{messageCount} new messages</span>}

      // Inline styles use JS objects — note double {{ }}
      <p style={{ color: 'blue', fontSize: 18 }}>Styled text</p>

      // Self-closing for elements with no children
      <img src="avatar.png" alt="User" />
      <input type="text" />

    </div>
  );
}

useState — Managing Component State

State is data that belongs to a component and can change over time. When state changes, React automatically re-renders the component to reflect the new data. The useState hook is how you add state to a function component.

You call useState(initialValue) and it returns an array with two things: the current value, and a setter function. You always destructure them: const [count, setCount] = useState(0). There is one critical rule that catches every beginner: never mutate state directly. Writing count = count + 1 changes the variable but React doesn't know about it and won't re-render. You must always call the setter: setCount(count + 1). The setter triggers a re-render with the new value.

Counter.tsx — useState explained in detailtsx
import { useState } from 'react';

function Counter() {
  // useState(0) sets the initial value to 0
  // count  → the current value (READ ONLY — never mutate directly!)
  // setCount → call this to update count and trigger a re-render
  const [count, setCount] = useState<number>(0);

  return (
    <div>
      <h2>Count: {count}</h2>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}

// ❌ WRONG — React won't re-render because it doesn't know count changed
// count = count + 1;

// ✅ CORRECT — calling the setter schedules a re-render
// setCount(count + 1);

useEffect — Running Code After Render

Some code doesn't belong in the render cycle — things like fetching data from an API, setting up subscriptions, or updating the page title. These are called side effects because they affect something outside the component's render output. The useEffect hook is how you handle them.

useEffect takes two arguments: a function to run, and a dependency array that controls when it runs. An empty array [] means "run once when the component first appears on screen" — perfect for initial data fetching. An array with values like [userId] means "re-run every time userId changes" — perfect for loading new data when a selection changes. No array at all means "run after every single render" which is rarely what you want.

UserProfile.tsx — fetching data with useEffecttsx
import { useState, useEffect } from 'react';

interface User { id: number; name: string; email: string }

function UserProfile({ userId }: { userId: number }) {
  const [user, setUser]       = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState<string | null>(null);

  useEffect(() => {
    setLoading(true);          // reset loading when userId changes
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(data => { setUser(data); setLoading(false); })
      .catch(() => { setError('Failed to load'); setLoading(false); });
  }, [userId]); // ← runs again whenever userId changes

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error: {error}</p>;
  if (!user)   return null;

  return <div><h1>{user.name}</h1><p>{user.email}</p></div>;
}

Rendering Lists with .map()

When you have an array of data and want to render a component for each item, you use JavaScript's .map() method inside JSX. This transforms an array of data objects into an array of JSX elements. There is one non-negotiable rule: every element in a list must have a unique key prop. React uses the key to track which items have been added, removed, or reordered between renders. Without keys, React may get confused and produce incorrect updates. Always use a stable unique identifier like a database ID — never use the array index as a key unless the list never changes order.

ProductList.tsx — rendering arraystsx
interface Product { id: number; name: string; price: number }

function ProductList() {
  const products: Product[] = [
    { id: 1, name: 'Phone', price: 999 },
    { id: 2, name: 'Laptop', price: 1499 },
    { id: 3, name: 'Headphones', price: 199 },
  ];

  return (
    <ul>
      {products.map(product => (
        // key must be unique and stable — use the database ID
        <li key={product.id}>
          <strong>{product.name}</strong> — {product.price} USD
        </li>
      ))}
    </ul>
  );
}

// Component 2: SearchableList
interface SearchableListProps {
  items: string[];
}

function SearchableList({ items }: SearchableListProps) {
  const [query, setQuery] = useState('');

  // .filter() returns only items that include the query (case-insensitive)
  const filtered = items.filter(item =>
    item.toLowerCase().includes(query.toLowerCase())
  );

  return (
    <div>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Search..."
      />
      <ul>
        {filtered.map((item, index) => (
          <li key={index}>{item}</li>
        ))}
      </ul>
      <p>{filtered.length} result(s)</p>
    </div>
  );
}

Vite — Why Your Dev Server Should Be Instant

For years, Webpack was the standard build tool. It works, but it has a fundamental design problem: it has to bundle your entire application — reading all files, resolving all imports, combining everything — before it can show you anything. On a large project this can take 30–60 seconds just to start the dev server. Every time you save a file, it re-bundles a large chunk of the app before the browser updates.

Vite (French for "fast") takes a completely different approach. Modern browsers support ES Modules natively — they can import files directly from the server one by one. Vite exploits this: the dev server starts instantly (it doesn't bundle anything), and when you request a page, it serves only the files that page actually needs, transforming them on-demand. When you save a file, only that file is re-sent — not the whole app. This is called Hot Module Replacement (HMR) and it's what makes modern development feel magical: you save a file and the browser updates in milliseconds, without losing your app's current state.

For production, Vite uses Rollup under the hood to bundle everything into optimized files — tree-shaken, minified, and split for efficient loading. You get the best of both worlds: instant development and optimized production builds.

vite.config.ts — fully annotatedts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';  // JSX transform + fast refresh
import path from 'path';

export default defineConfig({
  plugins: [react()],            // enables React JSX and Fast Refresh

  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src')
      // now: import Button from '@/components/Button'
      // instead of: import Button from '../../components/Button'
    }
  },

  server: {
    port: 3000,       // dev server URL: http://localhost:3000
    open: true,       // auto-open browser when you run npm run dev
    proxy: {
      '/api': 'http://localhost:8080'
      // requests to /api/users → forwarded to backend server
      // avoids CORS errors during development
    }
  },

  build: {
    outDir: 'dist',   // output folder for npm run build
    sourcemap: true   // include source maps for debugging production
  }
});

ESLint — Your Automated Code Reviewer

ESLint is a static analysis tool — it reads your code without running it and flags problems. Think of it as a tireless colleague who reviews every line you write and immediately points out: unused variables you forgot to remove, any types you accidentally left in, React hook calls that violate the rules of hooks, and dozens of other patterns that indicate bugs or code smell.

ESLint is configured via a rules file, and you can extend pre-built "rulesets" — collections of rules vetted by experts. The TypeScript ruleset catches type-related issues; the React hooks ruleset enforces that hooks are only called at the top level of components (a requirement React has for hooks to work correctly). You can set rules to "warn" (highlight in yellow but don't fail) or "error" (fail your CI pipeline).

.eslintrc.jsonjson
{
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended",
    "plugin:react-hooks/recommended"
  ],
  "rules": {
    "no-console": "warn",
    "@typescript-eslint/no-explicit-any": "error",
    "@typescript-eslint/no-unused-vars": "error",
    "react-hooks/rules-of-hooks": "error",
    "react-hooks/exhaustive-deps": "warn"
  }
}
💡
ESLint vs Prettier

ESLint catches code quality issues — things that could cause bugs or make code hard to maintain. Prettier only handles formatting — whitespace, line breaks, quote style. They don't overlap, and you should use both.

✅
Run ESLint in CI

Add "lint": "eslint src" to your scripts and run it in your CI pipeline so broken code can never be merged.

Prettier — End the Formatting Wars

Prettier is an opinionated code formatter. "Opinionated" means it makes its own formatting decisions and applies them consistently — you configure a few high-level options (tabs or spaces, print width, semicolons), and Prettier handles everything else. You stop thinking about how to format code and just write it; Prettier fixes it automatically on save.

The real value of Prettier is in teams. Without it, code reviews get filled with comments about formatting — two-space vs four-space indentation, trailing commas, quote style. These debates waste time and cause friction. With Prettier, the entire codebase looks like one person wrote it, automatically, forever. Once configured, you should never manually format code again — just let Prettier handle it.

.prettierrcjson
{
  "semi": true,          // semicolons at end of lines
  "singleQuote": false,   // use double quotes
  "tabWidth": 2,         // 2-space indentation
  "trailingComma": "es5",// trailing commas where valid
  "printWidth": 80,      // wrap lines at 80 chars
  "arrowParens": "always" // always (x) => not x =>
}
Before Prettier
// different styles everywhere
const x=1
function foo (a,b){
return a+b}
After Prettier
// consistent, readable
const x = 1;
function foo(a, b) {
  return a + b;
}

Vitest — Testing Your Components

Testing means writing code that checks your code works correctly. This sounds circular, but it's genuinely valuable: tests let you change your code with confidence. If you refactor a function and all tests still pass, you know you didn't break anything. If tests fail, you know immediately and exactly what broke — without having to manually click through the entire app.

Vitest is a testing framework built specifically for Vite projects. It's compatible with the Jest API (the most popular testing framework) so most Jest tutorials and docs apply directly. Combined with Testing Library, you can render React components and interact with them just like a user would — clicking buttons, filling in inputs, and asserting that the right text appears on screen. This approach ("test behavior, not implementation") means your tests remain valid even when you refactor the internals.

Toggle.test.tsx — testing by user behaviortsx
import { render, screen, fireEvent } from '@testing-library/react';
import { describe, it, expect, vi } from 'vitest';
import { Toggle } from './Toggle';

describe('Toggle component', () => {

  it('shows "Turn On" by default', () => {
    render(<Toggle />);
    // getByText finds an element with this text — fails if not found
    expect(screen.getByText('Turn On')).toBeInTheDocument();
  });

  it('shows "Turn Off" after clicking the button', () => {
    render(<Toggle />);
    fireEvent.click(screen.getByText('Turn On')); // simulate click
    expect(screen.getByText('Turn Off')).toBeInTheDocument();
  });

  it('calls an onToggle callback with the new state', () => {
    const handleToggle = vi.fn(); // mock function tracks calls
    render(<Toggle onToggle={handleToggle} />);
    fireEvent.click(screen.getByRole('button'));
    expect(handleToggle).toHaveBeenCalledWith(true);
  });

});

Here is the complete setup process for a professional React + TypeScript project from scratch. Every step below is something you'll repeat many times in your career.

Scaffold with Vite

2
Install dependencies

3
Install ESLint and Prettier

Run npm install -D eslint prettier @typescript-eslint/eslint-plugin eslint-plugin-react-hooks. Create config files for each.

4
Add testing

Run npm install -D vitest @testing-library/react @testing-library/jest-dom. Add "test": "vitest" to your scripts.

5
Create .gitignore

Add node_modules/, dist/, .env, and .DS_Store. This ensures you never accidentally commit generated files.

6
Start developing

Run npm run dev. Your app is live at http://localhost:5173 with hot module replacement enabled.
